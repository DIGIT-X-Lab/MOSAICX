MOSAICX Extraction Branch Atlas

Where the extraction path splits, expands, and gets risky.

This page maps the current extraction architecture as it actually behaves in the code after the planner simplification. The emphasis is practical: which branches are real hot-path decisions, which fallback ladders remain, and which pieces still look expensive or fragile enough to cut.

Two runs are wired in from the investigation: the full PDF with --force-ocr through PPStructure, and the Chandra-produced Markdown passed back into mosaicx extract.

Observed Facts

  • Full PDF + PPStructure: 4170 chars, full_text_default, route heavy_extract, outlines_fast succeeded first try.
  • Chandra Markdown: 2550 chars, short_doc_bypass, route heavy_extract, outlines_fast succeeded first try.
  • OCR quality changed field quality materially before any structured fallback was even needed.
  • ReAct long-doc planning is removed from the extraction hot path.
  • Default saved output still hides _planner and _source unless ZENTA is enabled.

Selected Component

full_text_default

Current long-document planner behavior: keep full section text and avoid a planner LLM in the extraction hot path.

Why it exists

Preserve correctness on longer documents by removing the risky planner compression step.

What it changed in our runs

In the current PPStructure PDF run, the only section stayed on heavy_extract with the full OCR text preserved. The Chandra Markdown run still used short_doc_bypass because the document was shorter.

Risk if kept

Higher token usage on long documents, and the standard-mode fallback ladder can now trigger more often because long-doc routes are openly marked as heavy_extract.

Risk if cut

Cutting this would mean reintroducing either hidden truncation or another planner heuristic. That would be a regression in predictability.

Current take

Keep. The bigger remaining cleanup targets are the standard-mode ladder, hidden diagnostics stripping, and repair bookkeeping when Refine is disabled.

Already removed: ReAct planner in the hot path Long documents now default to full_text_default and heavy_extract instead of planner-driven truncation.
Second pressure cut: standard-mode overlap BestOfN, adjudication, typed DSPy, adapters, and JSON rescue are too many overlapping fallback concepts for a clean SDK story.
Operational cut: hidden diagnostics stripping Developers should not need a special env flag to inspect planner and provenance behavior.
Cleanup cut: repair stage when Refine is disabled Right now it can record a repair need without doing a repair attempt. That is useful for logging, but muddy in the mainline path.
